# 玉米顾客购买与入槽动画设计

## 范围

修复两个玉米区行为：

1. 玉米区顾客必须从所属玉米田的出售槽购买玉米，完成装载后按森林顾客相同的流程结算并离开。
2. 玩家、搬运工或其他玉米生产者把玉米送入收集槽、携带槽和出售槽时，必须播放与森林库存相同语义的入槽动画，而不是瞬移到堆叠位置。

森林 `NPCScheduler`、森林 `StoragePoint`、森林顾客和木材购买流程保持不变。

## 架构

新增独立的 `CornCustomerScheduler`。它复制森林顾客调度器的生成、排队、移动、购买进度、顾客装载、结算、金币掉落和离场状态，但出售库存只接受 `CornStoragePoint`，并且只解析自身所属 `Finish` 或 `Finish-001` 下的 `Sell1`。左右玉米田的调度器分别持有各自出售槽，不能回退到森林中央出售槽，也不能读取另一块玉米田的库存。每位顾客买满四个玉米后固定生成三枚金币，与木头区当前 `coinReward = 3` 完全一致。

玉米顾客实例可以克隆现有顾客外观预制体，但克隆后必须停用其中的森林玩法组件。顾客携带槽由玉米调度器建立或迁移为独立 `CornStoragePoint`；森林 `StoragePoint` 不参与玉米购买转移。

扩展 `CornStoragePoint` 的动画实现，使动画类型与森林存放区保持相同语义：类型 1 为短距离落下，类型 2 为先升高再弹落，类型 3 为夸张旋转落下，类型 4 为搬运转移使用的弧线移动并在结束后解锁资源。动画代码保留在玉米脚本内，不调用森林 `ResourceManager` 或森林 `StoragePoint`。

## 数据流

玩家进入玉米出售区后，`ResourceFieldSystem` 从 `MultiResourceBackpack` 取出对应资源，并调用所属 `CornStoragePoint.addResource(item, 2)`。出售槽先记录库存和目标堆叠位置，再从物品当前世界位置播放升高、下落和弹跳动画。

玉米顾客到达购买点后，`CornCustomerScheduler` 读取本田 `Sell1` 下的 `CornStoragePoint`。每次购买调用玉米库存自己的 `removeResource(4)` 和顾客携带库存的 `addResource(..., 4)`，更新需求气泡与进度。顾客装满四个玉米后，执行与森林顾客相同的完成动画、队列推进和离场流程，并按 0.1 秒间隔创建三枚金币。金币使用森林区相同的金币预制体、堆叠坐标、掉落动画和 `coinDropArea` 金币库存容量检查，使玩家能够按原有流程拾取。

搬运工从田内收集槽向出售槽转移玉米时继续使用类型 4。资源在动画结束前保持不可再次移动；动画完成或应用从后台恢复后，库存恢复机制会把资源标记为可移动，避免购买链永久卡住。

## 场景接线

`DevScene` 中左右两个 `NPCScheduler-001` 节点的森林 `NPCScheduler` 组件迁移为 `CornCustomerScheduler`，保留已有路径点、顾客预制体、购买提示、金币掉落点和时间参数，并把左右 `coinReward` 都设置为 3。迁移后场景中只有森林中央顾客节点继续使用 `NPCScheduler`。

玉米调度器在启动时解析本模块根节点下的 `Sell1` 和其中的 `CornStoragePoint`。缺少出售槽、顾客携带槽或金币掉落绑定时，停止对应购买循环并输出一次明确错误，不创建森林库存作为备用。

## 验证

回归测试必须覆盖：

- 两个玉米顾客调度节点序列化为 `CornCustomerScheduler`，森林中央调度器类型不变。
- 玉米调度器只引用 `CornStoragePoint`，不存在对 `NPCScheduler`、森林 `StoragePoint` 或 `ResourceManager` 的导入和组件调用。
- 左右顾客分别解析所属 `Sell1`，不存在中央森林出售槽回退。
- `CornStoragePoint` 对动画类型 1、2、3、4 都建立补间，类型 4 只在完成回调后将资源标记为可移动。
- 玩家入槽仍使用类型 2，搬运和顾客购买仍使用类型 4。
- 顾客装满四个玉米后执行结算、金币掉落和离场状态；每单严格生成三枚金币，且金币库存已满时停止超量生成。
- 完整 Node 回归、场景 JSON 解析、本轮 TypeScript 诊断和 `git diff --check` 通过。

最终必须在 Cocos Creator 中目视确认：玉米飞入出售槽、顾客逐个取走四个玉米、需求气泡递减、每位完成购买的顾客生成三枚金币，并且左右玉米田库存互不串用。
