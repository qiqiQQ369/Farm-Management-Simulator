# 玉米区 HTML Release 运行时修复设计

## 目标

只修复 Cocos Creator 构建出的 HTML Release 中第二关玉米区异常，不改变编辑器预览、本地运行和森林区现有行为。

需要同时保证：

- 解锁玉米工人后稳定生成三名工人，后续解锁拖拉机和搬运工时工人仍然存在。
- 玉米产物进入售卖槽时位置和动画正确。
- 入槽动画完成后，玉米立即处于可购买状态。
- 即使网页端动画回调被打断，顾客也能在短暂恢复后继续逐个购买玉米。

## 已确认的 Release 风险

第二关大量对象在运行时创建，而森林区主要使用场景中已经序列化的固定节点。当前玉米区还存在以下不稳定依赖：

1. 工人从 inactive 的场景模板子节点克隆，模型后代节点的激活状态没有在生成完成时统一恢复。
2. 解锁拖拉机时会主动关闭已经生成的全部玉米工人，与三个生产角色同时工作的目标冲突。
3. 玉米售卖存储点由代码临时创建，玉米顾客随后再通过节点层级搜索它；两者之间没有显式绑定。
4. 玉米能否被顾客取走依赖入槽 Tween 的结束回调。网页端一旦中断回调，存储数量仍增加，但资源会永久停留在不可移动状态。

这些路径在编辑器预览中通常按预期顺序执行，在压缩后的 HTML Release 中则没有可靠的生命周期契约。

## 设计

### 玉米工人生成链

`ResourceFieldSystem` 仍负责生成三名玉米工人，但生成过程改为确定性步骤：

1. 克隆玉米工人视觉模板。
2. 清理继承的森林玩法组件。
3. 显式恢复根节点、模型节点、渲染器和动画组件。
4. 添加并启用玉米区独立的 `CornWorker`、碰撞体和刚体。
5. 设置出生位置、田垄目标和朝向后再激活节点。
6. 对缺失模型、动画、目标田垄的情况输出一次明确错误，不能静默生成空节点。

拖拉机解锁只新增拖拉机，不再关闭玉米工人。森林区工人和森林区脚本不做任何改动。

### 玉米售卖链

每块玉米田只允许存在一个名称确定的售卖存储节点。`ResourceFieldSystem` 创建并保存该 `CornStoragePoint` 后，通过显式配置接口传给对应的 `CornCustomerScheduler`。顾客不再依赖递归搜索来猜测售卖存储点。

玉米进入售卖槽时：

1. 记录入槽前世界坐标。
2. 挂到唯一售卖存储节点。
3. 恢复正确世界起点，再按售卖槽局部堆叠坐标播放动画。
4. 动画正常结束时标记为可移动。
5. 若 Tween 被停止或回调未执行，存储恢复机制在限定时间后把该玉米标记为可移动，并校正到目标槽位。

顾客只读取显式绑定的售卖存储。每次取走一个玉米后，存储数量、资源表和移除表必须保持同步。

## 验证边界

采用两个行为测试接口：

- 工人生命周期：完成工人解锁后得到三名 active 工人；完成拖拉机解锁后仍为三名 active 工人。
- 售卖事务：玉米入槽后可移动并能被顾客取走；模拟动画未完成时，恢复流程也能让顾客继续购买。

此外执行：

- 全部现有 Node 回归测试。
- Cocos Creator `debug=false` Web Mobile 构建。
- 检查 Release 包包含玉米工人、售卖存储和顾客购买关键路径。
- 通过本地 HTTP 运行最终 HTML 包，检查控制台和资源请求。

## 不在本次范围

- 不调整森林区逻辑。
- 不改变玉米工人、拖拉机、搬运工的数值和路线。
- 不修改已经正常的编辑器预览表现。
- 不把编辑器自动产生的无关场景改动混入代码提交。
