# 玉米模型、工人路线与拖拉机边界设计

## 范围

修复 `DevScene` 中玉米田的三项行为：

1. 玉米被收获后，重生计时结束时必须恢复可见模型。
2. 仅玉米田解锁的三名工人按各自分配的田垄顺序收割，向前推进到田垄边界后反向继续。森林区工人的现有行为保持不变。
3. 玉米拖拉机只能在实际种有玉米的田地区域内行驶。

## 架构

`CornFieldProduction` 继续是玉米可收获状态、产出和重生计时的唯一管理者。每株玉米保留其场景根节点和任何运行时创建的模型子节点。收获时隐藏根节点；重生时恢复根节点及所有模型子节点的激活状态，将根节点还原到记录的完整缩放，再执行既有的生长补间。这样模型显示和可交互状态始终处于同一生命周期。

玉米区使用独立的 `CornWorker` 状态机。它复制森林工人的空闲、移动、砍伐和等待循环，但只接收一条玉米田垄的目标；每名工人从本田垄首株出生，沿 Z 轴直线推进，到田垄端点后仅在本田垄内折返。森林 `Woodcutter` 不接收玉米目标，树木逻辑不变。

玉米区使用独立的 `CornTractor`、`CornHauler`、`CornStoragePoint` 和 `CornUnlockPad`。它们分别复制森林拖拉机的路径状态机、搬运工的装卸状态机、库存堆叠与中断恢复机制、解锁点的扣币与进度显示机制，但只连接玉米生产、玉米库存和玉米解锁点。左右玉米存放节点在场景中直接序列化 `CornStoragePoint`，不挂载森林 `StoragePoint`。每个玉米田的拖拉机起点与终点由田垄最外侧的有效玉米位置推导并夹紧到玉米田边界内。

玉米区可以克隆森林区的角色或车辆节点作为外观模板，但克隆完成后必须立即递归停用模板自带的森林玩法组件，再单独挂载并启用对应的 `Corn*` 组件。尤其拖拉机不能同时运行模板内的 `LoggingTruck` 与 `CornTractor`，否则两个状态机会同时修改同一节点的位置和朝向。

## 数据流

收获玉米会设置该植株的重生截止时间并隐藏植株根节点。截止时间到达后，`CornFieldProduction` 会在模型从生长缩放恢复到记录的完整缩放之前，恢复场景根节点及模型子节点的可见状态。短暂的保护时间内，现有可用性检查仍会阻止玩家、工人和拖拉机再次收获。

玉米工人解锁时，`ResourceFieldSystem` 按田垄顺序将一条完整田垄分配给一个 `CornWorker`。每名工人只从自己当前方向的目标序列中选择玉米，跳过暂时不可收获的玉米；只有到达所属田垄的末端时才反向。目标回调仍由 `CornFieldProduction` 执行扣除和产出，绝不登记森林树木砍伐，也绝不写入木材仓储。

拖拉机解锁时，`ResourceFieldSystem` 先停用克隆节点内的森林玩法组件，再向唯一启用的 `CornTractor` 传入受玉米边界限制的田垄端点。拖拉机保持移动、转向和接触收割循环；接触收割仍委托给 `CornFieldProduction`。玉米搬运工使用 `CornHauler` 将 `CornStoragePoint` 中的玉米由田内收集点搬到玉米出售点；三个解锁点使用 `CornUnlockPad`，通过玩家已挂载金币的通用接口扣币并更新自身进度。进度填充按 `Sprite.Type.FILLED` 识别，以兼容场景中名为 `splash2` 的填充节点。

## 异常处理

玉米田没有有效植株时，工人不接收目标，拖拉机也保持禁用，不提供无边界的备用路线。植株根节点失效时，在显示恢复与目标选择中跳过它。模型子节点若曾在收获期间被禁用，重生时只恢复已标记为该植株可视模型的节点。

## 验证

回归检查需要验证：重生生命周期会在生长补间前恢复可视子节点；玉米工人按照田垄顺序并在边界反向，且不改变森林区树木目标分支；玉米拖拉机、搬运工、存放区与解锁点均为专用组件；场景中的玉米存放组件类型必须是 `CornStoragePoint`；克隆模板的森林玩法组件必须在 `Corn*` 组件启动前停用；解锁进度必须更新 FILLED Sprite；拖拉机端点根据玉米植株边界计算并夹紧。现有玉米资源隔离测试必须继续证明玉米产出不会进入森林木材仓储。完整 Node 回归测试、场景 JSON 解析、TypeScript 诊断和 `git diff --check` 都必须通过。
